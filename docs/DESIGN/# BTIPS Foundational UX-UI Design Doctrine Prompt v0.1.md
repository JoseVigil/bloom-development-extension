# BTIPS Foundational UX/UI Design Doctrine Prompt v0.1

## Propósito del Documento

Este documento define la doctrina fundacional de diseño UX/UI del sistema BTIPS. No describe simplemente una interfaz sino un entorno cognitivo completo. El objetivo es establecer una base conceptual estable que permita diseñar el workspace, onboarding y superficies funcionales sin desviaciones filosóficas ni inconsistencias conceptuales.

BTIPS no es una aplicación convencional. Es un entorno cognitivo persistente orientado a la continuidad técnica y organizacional.

Cada decisión de diseño debe reforzar la percepción de continuidad, estabilidad y claridad cognitiva.

El sistema debe transmitir que nada se pierde y que todo evoluciona.

---

# BTIPS como Cognitive Operating Environment

BTIPS debe diseñarse como un Cognitive Operating Environment.

No es un IDE.
No es una AI tool.
No es un gestor de proyectos.

Es un entorno donde el pensamiento técnico ocurre y persiste.

El usuario no debe sentir que abre una aplicación.

Debe sentir que entra en su entorno de trabajo permanente.

El sistema debe transmitir estabilidad.

El usuario debe percibir que el sistema guarda su historia técnica y organizacional de forma continua.

BTIPS representa continuidad cognitiva.

El usuario debe sentir que el sistema crece con él.

---

# Principio de Continuidad

La continuidad es el valor central del sistema.

Todo debe diseñarse para transmitir continuidad:

* La información nunca se pierde.
* Las decisiones permanecen.
* El conocimiento crece.
* Los Mandates quedan registrados.
* Los Intents forman historia.

El usuario debe percibir que BTIPS es un sistema que acumula conocimiento.

Nada debe parecer efímero.

Nada debe parecer temporal.

Todo debe sentirse estable.

---

# Single Cognitive Workspace Window

El Workspace debe existir como una única ventana principal.

Esta ventana representa el centro del trabajo cognitivo.

El usuario debe poder pasar horas dentro de esta ventana sin fatiga ni confusión.

El Workspace no debe fragmentarse en múltiples ventanas principales.

El Workspace representa el lugar donde ocurre el pensamiento consciente.

Toda la navegación debe sentirse como permanencia dentro del mismo entorno.

El usuario no debe sentir que navega.

Debe sentir que permanece.

---

# Workspace como Centro Cognitivo

El Workspace Electron es el centro de gravedad del sistema.

Es el lugar donde:

* Se crean Mandates
* Se ejecutan Intents
* Se gobierna el Nucleus
* Se mantiene la continuidad

El Workspace es el lugar donde el usuario piensa.

No es solo una UI.

Es el centro cognitivo.

---

# Distributed Cognitive Surface

BTIPS funciona como un sistema distribuido de superficies cognitivas.

El Workspace Electron es el centro.

Pero existen superficies complementarias:

* VSCode
* Perfiles Chromium
* Landing
* Discovery

Estas superficies no son externas al sistema.

Son extensiones funcionales.

El usuario debe poder interactuar con BTIPS desde cualquiera de estas superficies sin perder continuidad.

Esto reduce fricción cognitiva.

El sistema debe sentirse presente incluso cuando el Workspace no está abierto.

---

# Persistent Profile Anchor

Cada perfil Chromium debe contener una Landing persistente.

La Landing funciona como ancla cognitiva.

El usuario siempre debe poder volver a Landing.

Landing representa identidad y continuidad.

Si el usuario se pierde puede volver a Landing.

Landing debe transmitir estabilidad.

Debe sentirse como el hogar del sistema.

El Workspace es el taller.

Landing es el hogar.

---

# Functional Web Surfaces

Las páginas BTIPS no son páginas web tradicionales.

No son marketing.

No son dashboards genéricos.

Son superficies funcionales del entorno cognitivo.

Estas páginas deben diseñarse con los mismos principios que el Workspace.

El usuario no debe sentir que salió del sistema.

Debe sentir que cambió de superficie.

---

# Visual Continuity Across Surfaces

Todas las superficies deben compartir un mismo lenguaje visual.

No necesariamente el mismo CSS.

Pero sí:

* Tipografía coherente
* Espaciado coherente
* Contraste coherente
* Densidad coherente

El objetivo es continuidad cognitiva.

El usuario debe sentir que sigue dentro del mismo sistema.

Nunca debe parecer que entró a una web externa.

---

# Deep Work Visual Calm

BTIPS debe diseñarse para sesiones largas.

El sistema debe producir calma visual.

Debe evitar ruido visual.

Debe evitar saturación.

Debe evitar contrastes agresivos.

Debe permitir horas de uso continuo.

El usuario debe sentir tranquilidad.

El sistema debe parecer silencioso.

Debe parecer estable.

Debe parecer confiable.

El usuario debe sentir:

"Aquí puedo pensar."

---

# Paleta Visual Calm-First

La paleta visual no debe diseñarse con objetivos de branding.

Debe diseñarse con objetivos cognitivos.

La prioridad es la calma visual.

Los colores deben reducir fatiga visual.

El contraste debe ser moderado.

El dark mode debe ser base.

El sistema debe evitar colores brillantes extensos.

Los colores intensos deben usarse solo como acentos.

El sistema debe parecer neutro.

---

# Modelo Mandate → Intent

La estructura del sistema es:

Mandate → Intent

Un Mandate representa una misión cognitiva.

Un Intent representa una acción cognitiva.

Esta estructura debe ser simple.

Debe ser comprensible sin entrenamiento.

El usuario debe entenderla intuitivamente.

Eliminar complejidad innecesaria es un principio fundamental.

---

# Mandate como Contenedor Cognitivo

Un Mandate no es un tab común.

Un Mandate es un universo de trabajo.

Cada Mandate representa una misión.

El usuario debe sentir que entra en un espacio significativo.

Los Mandates deben sentirse estables.

No efímeros.

No descartables.

---

# Tabs de Mandate

Los Mandates deben representarse como tabs dinámicos.

Cada Mandate abre un tab.

Los tabs deben permitir:

* Abrir Mandates
* Cambiar Mandates
* Cerrar Mandates

Cada tab debe mostrar:

1 Qué es el Mandate
2 En qué estado está
3 Si requiere atención

Esto debe ser visual.

No textual.

---

El sistema debe utilizar un indicador visual compacto.

▌▌▌

Este indicador representa:

* Estado
* Actividad
* Atención

Debe poder interpretarse en segundos.

Debe funcionar como lenguaje visual universal.

Debe minimizar necesidad de texto.

---

# Zero Friction Mandate Creation

La creación de Mandates debe ser inmediata.

Debe existir un botón "+" al lado de los tabs.

Este botón debe funcionar como en Chrome.

El usuario debe poder crear un Mandate sin entrenamiento.

Este es el camino natural.

---

# Orígenes de Mandate

Los Mandates pueden crearse desde:

Workspace (+)

VSCode (selección de archivos)

Store (instalación)

El sistema debe soportar los tres.

Pero el botón + es esencial para descubrimiento.

# Alfred como Asistente Local-Contextual

Alfred es el asistente del sistema.

Alfred no es un agente generalista.

Alfred es contextual.

Cada Mandate tiene su Alfred.

Alfred solo conoce el contexto activo.

Esto reduce ruido cognitivo.

---

# Alfred como Presencia

Alfred debe estar disponible siempre.

Pero no debe ser intrusivo.

Debe sentirse como una capa inteligente.

No como un chatbot separado.

---

# Ventanas de Alfred

Las conversaciones con Alfred deben poder abrirse como paneles.

Preferentemente abajo a la izquierda.

Cada Mandate puede tener su Alfred.

Los paneles pueden coexistir.

Cada panel debe tener identificación visual sutil.

Por ejemplo líneas de color.

El onboarding define el sistema inicial.

El onboarding debe ser estructurado.

No opcional.

No superficial.

El usuario debe completar onboarding para usar el sistema.

* El sistema contiene información
* El proyecto existe

Esto reduce cambio de contexto.
